What is Hiera?

Hiera is a built-in key-value configuration data lookup system, used for separating data from Puppet code. Master how Hiera keeps your Puppet code reusable while allowing it to behave differently for different nodes, environments, or teams.

❓ Why Does Hiera Matter?

Hiera eliminates hard-coded values and makes your Puppet code portable, maintainable, and environment-aware.

🔓

Separates Data from Code

Puppet's strength is in reusable code. Code that serves many needs must be configurable — you should put site-specific information in external configuration data files, rather than in the code itself.

🛡️

Eliminates Brittle Manifests

Without Hiera, you'd be forced to hard-code values directly in Puppet manifests. This makes your code brittle, environment-specific, and hard to maintain.

✍️

Write Once, Drive with Data

Hiera lets you write code once and drive behavior with data. Puppet uses Hiera to store configuration data in key-value pairs and look up what data a particular module needs for a given node during catalog compilation.

⚙️ How Does Hiera Work?

Four core concepts power Hiera's data lookup engine.

  1. The Hierarchy

    The hiera.yaml file configures a hierarchy: an ordered list of data sources. Hiera searches these data sources in the order they are written. Higher-priority sources override lower-priority ones. Most hierarchy levels use variables to locate a data source, so that different nodes get different data.

  2. Defaults with Overrides

    That's how Hiera's "defaults, with overrides" approach to data works — you specify common data at lower levels of the hierarchy, and override it at higher levels for groups of nodes with special needs.

  3. Lookup Methods

    Lookups are done via Automatic Parameter Lookup for classes included in the catalog, or through explicit lookup calls in your Puppet code.

  4. Supported File Formats

    Puppet comes with built-in support for JSON, YAML, and EYAML files. You can even extend Hiera to look up values in additional data sources, for example, a PostgreSQL database table, a custom web app, or a new kind of structured data file.

📝 Example: A Simple hiera.yaml Hierarchy

In this example, Hiera first looks for node-specific data, then OS-specific data, and finally falls back to common defaults — all without changing your Puppet code.

hiera.yaml
version: 5
hierarchy:
  - name: "Per-node data"
    path: "nodes/%{trusted.certname}.yaml"

  - name: "Per-OS data"
    path: "os/%{facts.os.name}.yaml"

  - name: "Common data"
    path: "common.yaml"

💡 How it resolves: When Puppet compiles a catalog for a node, Hiera walks down this hierarchy top-to-bottom. If a key is found in the node-specific YAML file it is used immediately. If not, Hiera continues to the OS-level file, then finally to common.yaml. This "defaults with overrides" pattern keeps your codebase clean and your data organized.

🌍 Real-World Use Cases

Hiera shines wherever configuration data varies across your infrastructure.

Use Case Description
Node-specific config Override settings for individual nodes (e.g., special DNS servers for a DB server)
Environment-based data Different values for dev, staging, and prod
Role / team-based config Apply different firewall rules for web vs. app servers
PE configuration management Manage Puppet Enterprise settings via Hiera

✅ When to Use Hiera

It's best to use Hiera when you want to:

1

Change Parameters in Non-Profile Classes

Override default class parameters without modifying module source code.

2

Set Static, Version-Controlled Parameters

Store values that rarely change in version-controlled YAML/JSON files alongside your code.

3

Configure for High Availability

Centralize HA-specific settings so they can be tuned per environment or node group.

🔍 Debugging Hiera

When things don't resolve as expected, Puppet gives you a powerful inspection tool.

puppet lookup can help identify exactly what Hiera was doing when it raised an error, or how it decided to look up a key and where it got its value. Use it during development and troubleshooting to trace the full resolution path through your hierarchy.

Shell — Basic Lookup
# Look up a single key for a specific node
puppet lookup profile::app::db_host --node app01.example.com

# Show the full resolution with explain flag
puppet lookup profile::app::db_host --node app01.example.com --explain
Shell — Explain Output (Abbreviated)
Searching for "profile::app::db_host"
  Hierarchy entry "Per-node data"
    Path "/etc/puppetlabs/code/environments/production/data/nodes/app01.example.com.yaml"
      Original path: "nodes/%{trusted.certname}.yaml"
      Found key: "profile::app::db_host" value: "db-primary.internal"
🐛

Diagnose Errors

Identify exactly which data source caused an error and see the raw value Hiera returned before Puppet attempted to use it.

🗺️

Trace Resolution Path

The --explain flag prints every hierarchy level Hiera checked, which paths existed, and where it ultimately found (or didn't find) the key.

✅

Validate Overrides

Confirm that your node-level or environment-level overrides are actually taking precedence over the common defaults.

🎯 Key Takeaways

Everything you need to remember about Hiera in four points.

📦

Data ≠ Code

Hiera separates site-specific configuration from your Puppet manifests so code stays reusable.

📐

Hierarchy-Driven

An ordered list of data sources lets you set defaults and override them at any level.

🔌

Extensible

Supports YAML, JSON, EYAML out of the box — and custom backends like PostgreSQL or web apps.

🔍

Debuggable

Use puppet lookup --explain to trace exactly how Hiera resolves every key.
