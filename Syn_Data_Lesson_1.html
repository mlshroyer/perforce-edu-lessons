Synthetic Data vs. Masking
vs. Tokenization

Modern enterprises face a critical tension: they need realistic data to develop, test, and innovate, but they must also protect sensitive information and comply with regulations like GDPR, HIPAA, and PCI DSS. This lesson defines each technique, explains how they differ, walks through focused examples, and provides guidance on when to use each.

1. What Is Synthetic Data?

Delphix Synthetic Data enables teams to generate realistic, production-like test data that does not exist in production. Unlike masking, which transforms existing production data, Synthetic Data creates entirely new datasets from scratch.

🧬

How It Works

Delphix Synthetic Data analyses the reference schema, examining table structures, column data types, constraints, relationships and semantics. Configure generation rules to define how synthetic values are produced for each table and column. Synthetic Data architecture is organized around applications, datasets, connectors, and jobs that work together to generate realistic synthetic data while preserving relationships between database objects.

  • No production data is involved
  • Data is entirely artificial — generated from metadata and rules, not derived from any real records
  • Supports testing scenarios that production data alone cannot cover
  • Generates realistic, statistically representative datasets for non-production environments

2. What Is Data Masking?

The masking capability of the Delphix DevOps Data Platform represents an automated approach to protecting non-production environments, replacing confidential information such as social security numbers, patient records, and credit card information with fictitious, yet realistic data.

🛡️

Key Characteristics

Unlike encryption measures that can be bypassed through schemes to obtain user credentials, masking irreversibly protects data in downstream environments.

  • Source: Real production data
  • Direction: One-way / irreversible
  • Output: Realistic, fictitious replacements
  • Consistent masking of data while maintaining referential integrity across heterogeneous data sources
  • No programming expertise required

3. What Is Tokenization?

Tokenization is a form of encryption where the actual data — such as names and addresses — are converted into tokens that do not convey any meaning (with regard to appearance and formatting). The Tokenization framework allows you to mask data and reverse its masking.

🔐

Key Characteristics

The algorithm tokenizes values using AES-128 encryption in CBC-CTS mode, with an optional initialization vector (IV), and Base64 encoding. The results are alpha-numeric strings that are longer than the original values.

  • Source: Real production data
  • Direction: Reversible (data can be re-identified)
  • Output: Encrypted tokens (alpha-numeric strings)
  • Mask data before sending to external vendors for analysis
  • Reverse the masking to take action on flagged accounts

5. Realistic Examples Demonstrating the Differences

Four practical scenarios illustrating when — and why — each technique is the right fit.

Example A

Testing a New Payments Feature — Synthetic Data

Scenario: A fintech company is building a new "Buy Now, Pay Later" installment feature. The production database has no installment plan records because the feature doesn't exist yet.

✔ Why Synthetic Data Fits

Synthetic Data helps development and testing teams create structurally valid, referentially consistent datasets for scenarios that production data cannot support, such as testing new schemas, edge cases, or early-stage applications. The team generates thousands of synthetic customer profiles, orders, and installment schedules — including edge cases like $0 installments, overdue payments, and maximum-term plans — without waiting for production data that does not yet exist.

✘ Why Masking Does Not Fit

There is no production data to mask. Masking transforms existing records; it cannot create records for a schema that has never held data.

Example B

Refreshing a QA Environment with Production-Like Data — Masking

Scenario: A healthcare organization needs to populate its QA environment with realistic patient data for regression testing of its claims processing system. The production database contains millions of real patient records with names, diagnoses, SSNs, and insurance IDs.

✔ Why Masking Fits

Delphix Masking is a multi-user, browser-based web application that provides complete, secure, and scalable software for your sensitive data discovery, masking, and tokenization needs. The team masks patient names, SSNs, and diagnoses with fictitious but realistic replacements. Referential integrity is preserved, so a masked patient's ID still links correctly across claims, encounters, and billing tables. The result is a fully functional, HIPAA-compliant test environment that mirrors the complexity and volume of production.

✘ Why Synthetic Data Does Not Fit

The organization specifically needs to test against the full complexity and data distribution patterns of its real production workload. Masked production data preserves these real-world distributions, relationships, and edge conditions as they actually exist, which is critical for regression testing.

Example C

Sharing Data with an External Analytics Vendor — Tokenization

Scenario: A retail bank sends transaction records to an external fraud-analytics vendor. After the vendor flags suspicious accounts, the bank must map those flagged tokens back to the original account holders to take action.

✔ Why Tokenization Fits

You can use a Tokenization algorithm to mask data before you send it to an external vendor for analysis. The vendor can then identify accounts that need attention without having any access to the original, sensitive data. Once you have the vendor's feedback, you can reverse the masking and take action on the appropriate accounts. The bank tokenizes account holder names and account numbers before export; the vendor performs its analysis on tokens; when results are returned, the bank re-identifies the tokens back to real account holders.

✘ Why Masking Does Not Fit

Masking is irreversible. If the bank masked account data before sending it to the vendor, there would be no way to map the vendor's flagged results back to the actual accounts.

✘ Why Synthetic Data Does Not Fit

The vendor needs to analyze patterns in real transaction data (volumes, frequencies, geographies). Entirely synthetic data would not reflect real fraud patterns in the bank's portfolio.

Example D

Edge-Case and Chaos Testing for an API — Synthetic Data

Scenario: A development team is building a new REST API for an insurance quoting engine and needs to test boundary conditions — policies with 100 riders, zero-premium quotes, quotes for customers at the maximum insurable age, and malformed input payloads.

✔ Why Synthetic Data Fits

Delphix Synthetic Data removes data bottlenecks by generating realistic, scenario-specific data on demand — eliminating data bottlenecks and enabling developers, testers, and AI agents to move at agentic speed. The team generates targeted synthetic records that include these extreme and negative scenarios, which simply do not exist in production.

✘ Why Masking Does Not Fit

Production data contains only "normal" policies. Masking those records would produce safe copies of normal data — it cannot introduce edge cases or boundary conditions that aren't already present.

6. When to Use Each Technique — Decision Guide

Match the right technique to the right circumstance using the guidance below.

🧬 Use Synthetic Data When…

  • Testing new schemas, edge cases, or early-stage applications where no production data exists
  • You need specific, tailored data characteristics — boundary values, negative scenarios, stress-test volumes
  • Compliance regulations prohibit any use of production data, even in transformed form
  • You want to accelerate development and testing cycles while supporting data privacy and compliance requirements

🛡️ Use Masking When…

  • You have mature production data and need to replicate its real-world complexity in non-production environments
  • You need a permanent, one-way transformation — masking irreversibly protects data in downstream environments
  • Regulations (GDPR, HIPAA, PCI DSS) require that sensitive data never appear in dev/test, but realistic structure and volume must be preserved
  • You need seamless integration of masking with data delivery capabilities before data is sent to an offsite data center or the public cloud

🔐 Use Tokenization When…

  • Data must be shared externally (vendors, partners) but you need to reverse the transformation afterward to act on results
  • You need to obfuscate data sent for processing, then reverse it when the processed data set is returned
  • Business workflows require round-trip data protection — tokenize → process → re-identify
