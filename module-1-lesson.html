Section 1

What Is the Akana API Portal?

1.1 Overview

The Akana API developer portal is a key component of the Akana API platform. It's more than a portal — it's a portal, marketplace, catalog, and directory all in one.

Think of it as the "front door" of your API ecosystem — a single, unified space where API producers, app developers, administrators, and business stakeholders all meet.

💡
Real-World Analogy: Imagine the Akana API Portal as an app store for APIs. Just like a mobile app store lets developers find, download, and use apps, the Akana Portal lets developers find, access, and consume APIs — complete with documentation, security policies, and usage analytics.

1.2 What Can You Do With It?

With Akana, you can make it easy to:

📤

Publish & Deploy

Publish and deploy consumable API products to make them available to your developer community.

🔍

Discover

Find existing API products and view their documentation to understand how to use the APIs.

🤝

Adopt & Share

Adopt and share APIs. Create teams and test different APIs in a secure environment with metrics and analytics.

🧪

Explore & Test

The Community Manager developer portal provides a platform for app developers to easily explore, test, and connect with different APIs.

1.3 Who Uses the Portal?

With Akana, you'll have one marketplace for all stakeholders — your developers, API admins/product owners, and API consumers. This helps you ensure consistency by maintaining a single user interface. What each stakeholder can see and do will depend on their role and permissions.

Role Primary Activity
App Developer Create apps, find APIs, request access
API Admin Manage APIs, approve access requests, monitor usage
Business Admin Manage organizations, policies, and users
Site Admin Configure the portal, manage users and login settings
Section 2

Core Concepts

2.1 APIs

An API (Application Programming Interface) is a set of rules and protocols that allows software applications to communicate with each other.

The Akana API Platform API supports activities relating to setting up, managing, and supporting app and API owners and all other platform users. It has many features and capabilities to support such activities as logging in and out, setting up apps and APIs, creating contracts between apps and APIs, creating discussions and reviews, commenting on discussions and reviews, and creating and managing groups.

  • APIs are the products being published and consumed.
  • API Admins can design or import APIs, add implementations, define deployment zones, fine-tune, test, and monitor — and manage all activities relating to the API and its connected apps.
  • Akana supports Swagger, OAS, RAML, WSDL from a single platform.
🔑
Key Term: An API in the portal represents both the technical service endpoint and the managed product — complete with documentation, security, and lifecycle governance.

2.2 Apps

An App in the Akana API Portal represents a consumer application that needs to connect to and use one or more APIs.

If you are an app developer, you can use the Community Manager developer portal to:

🛠

Create & Manage

Create and manage apps within the portal environment.

📖

View & Test

View different APIs, review API documentation, and test APIs — either on their own or with your app.

🔗

Connect

Connect to APIs that you want to use with your app and view analytics.

🔒

Secure

Secure your app with credentials (keys/secrets) used for authentication.

Key points about Apps:

  • Every app must be registered in the portal before it can call an API.
  • Apps have their own credentials (keys/secrets) that are used for authentication.
  • Apps are linked to APIs through Contracts (see next section).
💡
Real-World Example: A mobile banking app that calls a payment API would be registered as an App in the portal. The developer creates the app entry, configures credentials, and then requests access to the Payment API.

2.3 Contracts / Access

A Contract is the formal agreement that grants an App permission to call an API. This is the mechanism for controlled access.

After you have defined your app on the platform and chosen one or more APIs that include functionality you want to use, the next step is to connect your app to an API by requesting API access.

Contract Statuses

Status Description
Approved The API Access Request has been approved but is still pending activation by the API Provider.
Activated The API Access Request was approved and your app is connected to the API.
Access Pending The API Access Request is pending approval by the API Provider.

Sandbox vs. Live

At the Endpoint page, choose the implementation you want access to: Sandbox or Live. Requests for access to the Sandbox environment are often, but not always, set up for auto-approval. Requests for access to a Live implementation can sometimes be auto-approved, but a request may require additional review.

How to Request API Access

  1. Use the Search feature to find an API you're interested in using.
  2. Click through from the search results to the API Details page.
  3. From the Overview page for the API, click Access in the upper right-hand corner.
  4. The API Access Wizard launches at the first step: Select App.
  5. Specify the app/app version that you want the API connection for. Click Next.
  6. At the Endpoint page, choose the implementation — Sandbox or Live — and click Next.
  7. If policies are defined, you will see a list and can choose one or more policies. Click Next.
  8. If the Licenses page displays, select a license option.
  9. If there are any legal agreements associated with the API, review and accept each one.
  10. Click Save.

2.4 Tenant / Organization

A Tenant (also referred to as an Organization or Business) is the top-level administrative unit within the Akana API Portal. It provides an isolated environment for managing APIs, apps, users, and policies.

The Akana API Platform is composed of primary components, including Community Manager — a customized, secure API portal to make your APIs accessible to vendors and internal users, quickly onboard partners, provision APIs, and establish active social channels.

A business can own one or more APIs and apps, and must have at least one Administrator. The Business Administrator automatically has administrator rights over all the APIs and apps owned by the business as well as all the users who are part of the business.

Concept Description
Tenant The top-level environment; often maps to a company or department
Organization / Business A sub-unit within the tenant that owns APIs, Apps, and Users
Business Admin Manages all APIs, apps, and users within the organization
Root Level The top-most Business Admin scope — changes here affect all businesses
💡
Analogy: Think of a Tenant like a corporate headquarters, and Organizations as its departments. Each department manages its own APIs and apps, but the corporate HQ (root level) can see and govern everything.
Section 3

Navigating the Akana API Portal

The Akana API Portal offers a structured navigation layout. Understanding each primary section is key to working effectively in the platform.

3.1 Dashboard

The Dashboard is your personalized activity hub — the first page you see after logging in.

📊

Activity & Alerts

Displays recent activity, notifications, and alerts related to your APIs and Apps.

🔔

Subscriptions

Subscribing to an app allows users to receive news or updates relating to the app on their personal Dashboard.

⚡

Quick Access

Provides quick-access links to APIs, Apps, and Forum activity.

👤

Personalized

Visible to all users — content is personalized based on your role and subscriptions.

🧭
Navigation Tip: The Dashboard is your home base. Start here to get a quick pulse on the state of your APIs and apps before navigating deeper.

3.2 Admin (Administration)

The Admin section provides access to platform-level and organization-level configuration settings.

The following content is available at the root level, via More > Admin, for users with the Business Administrator role:

🌐

API Gateways

👔

Business Admins

🗺

Deployment Zones

🔑

Domains

💬

Forum

📜

Licenses

🏷

Model Categories

🔐

Scopes

The Site Admin is responsible for keeping the site running smoothly, performing activities such as configuring site settings, managing users and other Site Admins, customizing the user interface, uploading and managing site content such as the landing page, online help, and developer agreement, and configuring login domains and custom workflows.

⚠️
Access Note: The Admin section is only visible to users with Business Admin or Site Admin roles. Standard users and app developers will not see all Admin options.

3.3 Users

The Users section enables administrators to manage portal user accounts and their assigned roles.

Key activities in the Users section:

  • View all registered users in the tenant.
  • Assign or modify user roles (App Developer, API Admin, Business Admin, Site Admin).
  • Manage user access and permissions.
  • Review user activity and group memberships.

3.4 APIs

The APIs section is the central catalog for browsing, managing, and administering APIs.

From the API Admin perspective, activities include designing or importing an API, adding implementations and defining deployment zones, fine-tuning, testing, and monitoring, and managing all activities relating to the API and its connected apps.

Action Who Can Do It
Browse & search APIs All Users
View API documentation All Users
Add / Import a new API API Admin, Business Admin
Manage API implementations API Admin
Review connected Apps API Admin
Manage API contracts API Admin, Business Admin
Monitor API analytics API Admin, Business Admin

3.5 Apps

The Apps section is where developers create and manage their registered applications.

To view a summary of the API contracts for your app, go to Apps > My Apps > choose app > APIs page. All APIs that your app has contracts with are shown on the APIs page, along with the status of each contract.

Key activities in the Apps section:

  • Create and configure a new App.
  • View and manage API connections (contracts).
  • Configure App credentials (API keys / OAuth).
  • Monitor App-level analytics.
  • Promote an App to the next environment (e.g., from Sandbox to Production).
💡
Pro Tip: Always ensure your App has the correct credentials configured before requesting API Access. Before you can make a call to an API, you must add credentials for your app.
