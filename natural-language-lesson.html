Crafting Effective Natural Language Testing Prompts in Perfecto AI

Introduction: A New Way to Express What You Already Know

If you've spent years writing Selenium locators, crafting XPath expressions, or maintaining Appium scripts, the shift to writing test instructions in plain English can feel surprisingly disorienting. The skills that made you successful — precision, logic, deep knowledge of application behavior — haven't gone away. In fact, they become more valuable here. What's changing is the medium, not the mindset.

Traditional test automation requires technical expertise to write and maintain scripts. Perfecto AI democratizes test creation through natural language prompts that anyone can use. Teams describe what they want to validate in plain English, and the AI generates and executes the corresponding tests. The AI User Action Assistant can navigate the device screen like a user would. Phrase your prompts like you would when describing a test step to a manual tester.

To have the test interact with the application, you no longer need to identify object locators or learn a scripting language. Think of it this way: instead of telling a machine exactly how to interact with the UI element by element, you're now describing the intent of the test — the way you might explain a test case to a colleague sitting next to you. Your deep understanding of what needs to be tested is the same; you're just communicating it differently.

Section 1: The Mindset Shift — From "How" to "What"

In script-based testing, you tell the tool how to do something. In natural language testing, you tell the AI what you want to accomplish.

Script-Based (How)

driver.findElement(By.xpath("//button[@id='confirm']")).click();

Natural Language (What)

Select VISA as payment method and confirm.

When you define test scripts traditionally as actions applied to locators, the test script breaks every time a UI element changes. Using the AI Assistant, you can describe the logical task in the same way as when talking to a manual tester.

This is a fundamental shift, and it's worth acknowledging that it may not feel intuitive right away — especially if your instinct is to be as technically precise as possible. That instinct is an asset. The adjustment is learning where to aim that precision: not at the element's technical identity, but at the logical goal of the interaction.

💡 Key Insight: Applications are designed and implemented to address human needs, not to address the needs of automated testing. AI validations work in a similar way: They first analyze the semantic and logical meaning of the requested validation and then try to match it with the screen content.

Section 2: The Five Characteristics of an Effective Prompt

Whether you're writing an AI User Action (an interaction with the app) or an AI Validation (a pass/fail check), effective prompts share five core characteristics.

Characteristic 1: Goal-Oriented, Not Mechanical

Describe what should happen, not how the UI should be manipulated.

❌ Don't Write✅ Do WriteWhy?
Move the mouse to the button labeled Cart in the bottom right corner On the profile page, open the shopping cart. Describe the logical goal. Do not refer to locators or labels that can change during development.

If you're coming from scripted automation, you've likely spent considerable time crafting precise locators. It's natural to want to include that level of specificity. But in natural language testing, the AI interprets what it sees on the screen, much like a human tester would.

The advantage of describing the logical task is that a manual tester — just as the AI Assistant — can follow the same instructions, even if the UI changes later. For example, the AI Assistant — just as a manual tester — can identify the button that confirms the selection and click it, no matter what it is called, or where on the screen it is.

Characteristic 2: Specific and Detailed Enough to Be Repeatable

Vague instructions produce unpredictable results. Provide enough context that the action can be performed consistently, every single time.

❌ Don't Write✅ Do WriteWhy?
Fill in the form Enter "Jill A. Smith" in the name fields. Enter Main Street 1 in NYC, USA, as the address. Don't be vague. Provide enough details for the AI Assistant to be able to repeat the task consistently.

For those accustomed to parameterized scripts, this is familiar territory — you always needed specific data. The difference now is that the data and the instruction live together in a single, readable sentence rather than being split between code and data files.

Characteristic 3: Unambiguous

When the screen contains multiple similar elements, your prompt needs to eliminate any doubt about which one you mean.

❌ Don't Write✅ Do WriteWhy?
Click the red shoes. From the search results, select the red sneakers that cost $30. Don't be ambiguous. Can it happen that the UI contains several similar elements? Use specific words.

Characteristic 4: Scoped Appropriately

For optimal performance, phrase each User Action so it does not resolve to more than 15 steps and so that it takes less than 3 minutes. A single prompt should represent one coherent user goal — not an entire end-to-end journey. Break complex flows into logical steps, just as you would break a test script into individual methods or steps.

Example — Filling Forms Efficiently:

When using AI User Actions to fill in forms, instead of filling each field as a separate action, write one prompt combining several value inputs. Then tell the Assistant to click the submit button as a separate action.

Less EfficientMore Efficient
Fill in username John ➜ Enter last name Doe ➜ Select year 2001 ➜ Submit the form Fill in username John, last name Doe, year 2001 ➜ Submit the form

Characteristic 5: Binary and Objective (for Validations)

This characteristic applies specifically to AI Validation prompts. A validation is a special type of question with a binary answer: yes or no, PASS or FAIL. When it comes to automated testing, a validation or assertion is the primary way to decide whether something is as expected.

❌ Not a Validation (Open-Ended)✅ A Proper Validation (Binary)
How much is the account balance? Is the account balance equal to $22?
What's the color of the button? Is the color of the login button green?
Whose name appears in the top left? Does the user name Joe appear in the app header?

If you've written assertions in code (e.g., assertEquals(balance, "$22")), you already think in binary terms. The same logic applies — you're simply expressing it conversationally now.

🔑 Pause & Reflect: Think about a test you've recently scripted. How would you express its key action steps and validation checks as plain-language sentences? Try writing two or three of them down before continuing.

Section 3: Avoiding Common Pitfalls

The openness of natural language is both its strength and its challenge. The free and open text used for visual validations is convenient but also opens the door for uncertainty, ambiguity, and other aspects that can reduce the robustness of your testing scenario.

Here are common pitfalls illustrated through a pizza restaurant menu scenario:

PromptProblemResult
"Does pizza taste good?" Too subjective — there's no objective answer on the screen Unreliable
"How does the pizza taste?" Irrelevant — not something the app can show Not a validation
"Does the pizza cost $80?" Too ambiguous — which pizza? The assistant concluded that the user asked about "any pizza."
"Does the Golden corn pizza cost $80?" ✅ Specific and objective Returns an accurate PASS

The lesson from script-based testing applies here too: the more precisely you define your expected outcome, the more reliable your test. The form of that precision has simply changed from code syntax to clear human language.

Section 4: The AI Assistant as Your Collaborative Partner

One of the most valuable aspects of this transition is that you don't have to get the prompt perfect on the first try.

The Assistant confirms by repeating the instructions as it understood them and adds them in the test editor. Verify that the Assistant interpreted the command as intended. If your input could be more specific, the Assistant asks for details or suggests an improved phrasing.

The Objectivity Score

The Assistant displays an objectivity score (green, yellow, or red) to indicate how robust your user action is phrased, and suggests paraphrases if necessary. Think of this as a real-time code review — except instead of reviewing syntax, it's reviewing the clarity and reliability of your natural language.

  • Green: Your prompt is clear and objective. Proceed with confidence.
  • Yellow: The prompt is functional but could be more robust. Consider the Assistant's suggestions.
  • Red: The prompt is too vague, ambiguous, or subjective. Rework before using in your test.

Iterative Refinement in Action

Here's an example of how a conversation with the AI Validation Assistant naturally sharpens a prompt:

StepWhoInput / Response
1 You "Does the pizza cost $80?"
2 AI PASS: There are pizzas that cost $80.
Suggested rephrasing: "Are there pizzas that cost $80 on the menu?"
3 You "And what about the Capsicum one?"
4 AI FAIL: The Capsicum pizza costs $140.
Suggested rephrasing: "Is the Capsicum pizza price equal to $80?"

The AI Validation Assistant crafts a suggestion that meets the entire semantic request from the discussion. The Assistant also transforms generic questions into a more objective validation format and corrects grammar and typos, making sure your test scenario looks great.

This iterative process may feel unfamiliar if you're used to writing code that either compiles or doesn't. But it's actually closer to how experienced testers naturally refine their understanding of requirements — through conversation and clarification.

Using Variables for Dynamic Data

To create a variable, describe its value and in which variable to store it in the format ${variable-name}. Use the variable in later steps, for example, in validations.

Identify the exchange rate shown and save it to ${rate}.
Get the value of the account balance and put it into ${balance}.
Log in as ${username} with password ${pwd}.

This makes your natural language prompts just as data-driven as parameterized scripts — without a single line of code.

Validation Prompts

What You'd Assert in CodeNatural Language Validation
assertEquals(getBalance(), "$22"); Is the account balance equal to $22?
assertTrue(isDisplayed(signUpLink)); Is there a sign-up option for new users labeled 'Sign Up Here'?
assertTrue(depositAmount > withdrawalAmount); Is the online deposit amount greater than the online withdrawal amount?
assertEquals(getLargestSegment(chart), "Savings"); Does the graph show Individual Savings as the biggest value?

Notice how the validations on the right side are closer to the business requirement than the code assertions on the left. With AI validations, validations are logical, closer to the business requirements, and can combine almost any aspects appearing on the screen.

Handling Complex Screens with Focus

To improve interaction precision with small or unclear elements, instruct the assistant to focus on a certain area. For example, use this more robust approach to select densely packed checkboxes, small text fields, or complex filter inputs. In these cases, tell the assistant first to "focus on" the section, to "look closely at…," to "zoom into…," to "look at the small text in…," or similar.

Example: Focus on the transfer history table and look at the first row from "internal fund transfer."

Handling Timing and Interruptions

In script-based testing, you'd write explicit waits (WebDriverWait, Thread.sleep). In natural language testing:

If your app is affected by irregular network latency, or a procedure is disrupted by a modal dialog, a hard-coded script can miss its cue. Instead, tell the AI Assistant to wait, to attempt to repeat an action until it is successful, to dismiss any modal dialog, or to wait for a progress bar to finish.

Example: Wait for the login button to appear, and then continue.

Section 6: Advanced Tip — Specialized Knowledge

Perfecto AI's general knowledge can reach its limits when testing applications with specialized UI elements, such as airplane seat plans, custom dashboards with diagrams, or canvases with drag-and-droppable elements.

You don't need to explain this specialized context as part of every input. Instead, write down your specialized knowledge in a plain text file.

📄 Example ai.txt content:

"The dashboard pie chart has a legend on the right-hand side. Percentages are hidden by default. Clicking on the pie charts displays the percentage as a number over the chart."

Save the file under the name ai.txt, and upload the ai.txt file to the root of the Perfecto Repository. The knowledge now appears in the AI Assistant as part of your request, as general user instructions.
