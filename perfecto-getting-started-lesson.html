Module 1

What Is Perfecto AI?

1.1 Overview

Perfecto offers AI-based features for test creation and test execution. The use of AI in software testing removes traditional testing barriers to help testing teams achieve new levels of agility, reliability, and breakthrough advancements. AI-based features enable testers of every skill level, allowing for simplified test creation, faster debugging, enhanced collaboration, and the elimination of test maintenance.

Perfecto AI, powered by Perforce Intelligence, delivers up to 70% efficiency gains with zero scripts, frameworks, or maintenance. This represents a fundamental shift: Perforce launched a breakthrough in agentic AI testing with the ability to turn plain language inputs into resilient, execution-ready test actions.

With one input, users can now validate applications across web, Android, and iOS — without object locators, code, or maintenance.

1.2 Where to Access Perfecto AI

You can access Perfecto's AI-based testing features through: AI commands in AI Scriptless, and AI-powered functions in your Appium scripts, available as part of Perfecto extensions. AI Scriptless allows you to automate mobile application testing without writing tests. You can design and execute tests using a visual interface, predefined actions, and AI-powered test generation.

📋 Licensing Note

The interface name and available features depend on your license configuration: If your cloud environment includes both the Perfecto AI license and the Desktop Web license, the interface is labeled AI Scriptless and supports test creation for both mobile and desktop web applications using natural language commands. If one or both of these licenses are not included, the interface is labeled Scriptless Mobile and supports mobile testing only.

1.3 The Three AI Commands

Use the following AI Scriptless AI commands when creating natural-language test steps that work independently of object locators, UI language, or dynamic content. You write the Perfecto AI-powered test once and can run it on all devices and operating systems.

AI CommandPurpose
AI Validation Assert or validate what appears on-screen using a natural language question with a PASS/FAIL outcome
AI User Action Perform user interactions (tap, type, navigate) described in natural language
AI Visual Comparison Compare two device screens like a human tester would, detecting visual differences

Module 2

Understanding the Paradigm Shift

2.1 Why Script-Based Tests Are Fragile

In traditional functional testing, tests are built on object locators (XPath, CSS selectors, accessibility IDs) tied directly to UI elements. When the UI changes — even cosmetically — these locators break and tests fail.

Testing continues to be a main blocker in DevOps and the automation of application delivery. Siloed tools, flaky tests, UI instability, test data bottlenecks, and triage delays create unsustainable costs and inefficiencies.

Many testing companies have turned to AI copilots, which generate scripts based on prompts. But those scripts still require frameworks, setup, and constant maintenance — often amplifying the very maintenance burden teams are trying to eliminate.

2.2 How Natural Language Changes Everything

The AI-based commands available in AI Scriptless and the AI functions available for use in Appium tests allow you to use natural language, in any language, just like in a conversation. This opens up new possibilities, such as writing complex validations that may combine multiple traditional commands into a single command or may even reach beyond what is possible with scripting, thereby expanding test coverage.

AI commands also allow you to overcome the fragility of traditional test scripts because they let you use different mechanisms for validating elements on the screen, are platform agnostic, and adapt to UI changes.

2.3 Think Like a Human, Not Like a Machine

❌ Traditional Approach

Traditional functional test automation uses and is aligned with actual elements in the GUI on the open device screen — buttons, text entry fields, drop-down menus. A test developer needs to assess the screen structure and create a logical meaning between different GUI elements and the required logic under test.

✅ Perfecto AI Approach

Unlike computer code, humans approach a device screen semantically: they break what they see down into logical meaning and match the logical, semantic understanding with visual representation. This is exactly how Perfecto AI works. AI validations use the same approach.

💡 Key Insight

Perfecto AI interprets screens the way a human does — semantically — rather than relying on brittle DOM structures. You describe what you see, not how the machine should find it.

Translating Script-Based Tests to Natural Language

This is where the real mindset shift happens. Below are side-by-side comparisons showing how traditional script-based test steps translate into natural language Perfecto AI steps.

3.1 Example: Login Validation (Banking App)

StepScript-Based Approach (Pseudo-Code)Natural Language (Perfecto AI)
Navigate driver.get("https://bank.example.com") AI User Action: "Open the banking app and navigate to the login page."
Enter username driver.findElement(By.id("username")).sendKeys("john_doe") AI User Action: "Enter 'john_doe' in the username field."
Enter password driver.findElement(By.id("password")).sendKeys("P@ss123") AI User Action: "Enter 'P@ss123' in the password field."
Click login driver.findElement(By.xpath("//button[@id='login-btn']")).click() AI User Action: "Click the login button."
Validate balance String bal = driver.findElement(By.xpath("//text[3]")).getText();
assertEquals(bal, "$15,678.62");		 AI Validation: "Is the available balance equal to $15,678.62?"

🔑 Key Insight

The script-based approach depends on precise locators (By.id, By.xpath) that can break when the UI changes. The natural language approach describes the intent — what a human tester would do and verify.

3.4 Example: User Actions — Form Filling

Instead of filling each field as a separate action, write one prompt combining several value inputs. Then tell the Assistant to click the submit button as a separate action.

Combine related field inputs into a single AI User Action for efficiency, but keep distinct interactions (like form submission) as separate steps for clarity and reliability.

Module 4

Rethinking Your Test Approach for Natural Language

The transition from scripting to natural language is not just a tooling change — it requires a shift in how you think about describing tests. Here are the critical considerations:

4.1 Questions vs. Validations

Make sure you understand the difference between a question and a validation:

  • A question (or query) is an open-ended question (what, who, where, when, why, how) where the answer can be any value from the device screen, and is not a validation.
  • A validation is a special type of question with a binary answer: yes or no, PASS or FAIL. When it comes to automated testing, a validation or assertion is the primary way to decide whether something is as expected.
❌ Open-ended Question (Not a Validation)✅ Proper Validation
"How much is the account balance?" "Is the account balance equal to $22?"
"What's the color of the button?" "Is the color of the login button green?"
"Whose name appears in the top left?" "Does the user name Joe appear in the app header?"

💡 Reflection Point

When you write a script-based assertEquals, you already know the expected value. In natural language, you must state that expected value explicitly in the form of a binary question.

4.2 Be Specific, Not Ambiguous

Asking the right questions is one of the biggest challenges with free and open text validations. Consider this pizza menu example:

PromptProblemResult
"Does the pizza cost $80?" Too ambiguous — which pizza? Unpredictable
"Is there a pizza that costs $80?" Clear — asks about any pizza ✅ PASS
"Do all pizzas cost $80?" Clear — asks about all ✅ NO (FAIL)
"Does the Golden corn pizza cost $80?" Clear — specific pizza named ✅ YES (PASS)

4.3 Describe the Goal, Not the Mechanics

When you define test scripts traditionally as actions applied to locators, the test script breaks every time a UI element changes. Using the AI Assistant, you can describe the logical task in the same way as when talking to a manual tester.

❌ Don't (Mechanical)✅ Do (Goal-Oriented)Why?
"Move the mouse to the button labeled Cart in the bottom right corner" "On the profile page, open the shopping cart." Describe the logical goal. Do not refer to locators or labels that can change during development.
"Fill in the form" "Enter 'Jill A. Smith' in the name fields. Enter Main street 1 in NYC, USA, as the address." Don't be vague. Provide enough details for the AI Assistant to repeat the task consistently.
"Click the red shoes." "From the search results, select the red sneakers that cost $30." Don't be ambiguous. Use specific words when the UI may contain several similar elements.

4.4 Leverage the AI Assistant to Refine Your Prompts

To get the most out of AI-based commands, when working with AI Scriptless, consider using the AI Assistant, at least in the beginning. The AI Assistant will respond to your input and may provide suggestions to help hone your input for the desired result.

🔄

Transforms Questions

Converts generic questions into more objective validation formats for reliable PASS/FAIL outcomes.

✏️

Corrects & Clarifies

Corrects grammar and typos to improve test readability and consistency across your test suite.

🌐

Multilingual Support

Works in almost any language. If English is not your language of choice, you may phrase the AI validation in any other language.

🎯

Objectivity Score

Displays an objectivity score (green, yellow, or red) to indicate validation robustness.

4.5 Write Once, Run Everywhere

Once you have fine-tuned the instructions, you can even run the same Scriptless test across a variety of devices and validate the same test steps on them. This saves time otherwise spent with repetitive test development and duplicated maintenance.

Since you do not refer to device-dependent UI elements, the AI Assistant interprets the instructions relative to what it sees on the device screen.

⚡ Cross-Platform Power

One natural language test — validated across web, Android, and iOS — without rewriting a single step.

4.6 Handle Dynamic Conditions Gracefully

If your app is affected by irregular network latency, or a procedure is disrupted by a modal dialog, a hard-coded script can miss its cue. Instead, tell the AI Assistant to wait, to attempt to repeat an action until it is successful, to dismiss any modal dialog, or to wait for a progress bar to finish.

❌ Hard-Coded Script

A fixed Thread.sleep(5000) or a brittle explicit wait tied to an XPath that may not exist yet.

✅ Natural Language

"Wait for the login button to appear, and then continue."

"Dismiss any pop-up dialog if one appears."

4.7 Keep Actions Manageable

⏱ Performance Guideline

For optimal performance, phrase each User Action so it does not resolve to more than 15 steps and so that it takes less than 3 minutes. Break complex workflows into logical, manageable steps.

Module 5

AI Visual Comparison — A New Testing Dimension

Beyond validations and user actions, Perfecto AI offers a capability that would be extremely complex to build in traditional scripts:

The AI Visual Comparison command compares two device screens like a human tester would. The assistant detects and describes visual differences in test result screenshots, and fails the test if the differences are in certain categories.

Detection Categories

🎨

Style

Font, color, and alignment changes between screenshots.

🔢

Value

Text or numeric data changes detected across screens.

🚫

Missing Elements

Elements present in the baseline but absent in the comparison screen.

↔️

Moved Elements

Elements that have shifted position between the two screens.

➕

Additions

New elements that appear in the comparison screen but not in the baseline.

⚠️

Errors

Overlapping elements, error messages, and other visual anomalies.

💡 Traditional vs. AI Visual Testing

This kind of visual regression testing would traditionally require pixel-comparison libraries with extensive custom configuration — now it's handled semantically by AI. The AI understands what changed and why it matters, not just that pixels differ.

Breaks when UI changes Adapts to UI changes automatically
Platform-specific test code Write once, run across web, Android, iOS
Requires coding expertise Accessible to testers of all skill levels

Key Principles for the Transition

🎯 Think in Outcomes Describe what you want to verify, not how to find it. Focus on outcomes, not DOM elements.
✅ Write Binary Validations Phrase everything as a yes/no, PASS/FAIL assertion. The AI needs a clear binary to evaluate.
🔍 Be Specific & Unambiguous Name the exact element, value, or condition you're testing. Eliminate room for misinterpretation.
🗣️ Describe Goals, Not Mechanics Say what a manual tester would understand, not what a machine would execute.
🤖 Use the AI Assistant Iterate with it to sharpen your prompts, especially when starting out. Let it guide your phrasing.
🧩 Keep Actions Atomic Break complex workflows into logical, manageable steps — no more than 15 sub-steps per action.

Practice Exercises

Apply what you've learned. Try each exercise, then click to reveal guidance.

Exercise 1: Convert Script to Natural Language

Given this script-based test step:

WebElement price = driver.findElement(By.xpath("//div[@class='product-price']/span[2]"));
assertEquals(price.getText(), "$49.99");

Your task: Write an equivalent Perfecto AI Validation in natural language.

💡 Reveal Hint & Guidance

Hint: Think about what the user actually sees and how you'd describe the check to a colleague.

Example Answer

"Is the product price equal to $49.99?"

Notice how you describe the visible intent — not the XPath path or DOM structure. If you can be more specific (e.g., name the product), even better.

Exercise 2: Write an Ambiguity-Free Validation

You are testing a flight booking app that shows departure and arrival times for multiple flights.

Your task: Write a natural language validation to confirm that Flight AA-204 departs at 3:45 PM.

💡 Reveal Hint & Guidance

Consider: What makes this specific enough for the AI to consistently return the correct answer?

Example Answer

"Does Flight AA-204 show a departure time of 3:45 PM?"

By naming the specific flight number and stating the exact expected value, you remove ambiguity — even if many flights are displayed on screen.

Exercise 3: Describe a Multi-Step User Action

Your task: Write a natural language AI User Action to complete a checkout process: select VISA as payment, enter card number 4111-1111-1111-1111, set expiry to 12/27, and confirm the purchase.

💡 Reveal Hint & Guidance

Remember: Describe the goal, provide specific details, and don't reference UI mechanics.

Example Answer

AI User Action 1: "Select VISA as the payment method. Enter card number 4111-1111-1111-1111 and set the expiry date to 12/27."

AI User Action 2: "Confirm and complete the purchase."

Grouping the payment detail entry as one action and the confirmation as a separate step follows the atomic actions principle while keeping the workflow readable.
