Puppet Anti-Patterns

What They Are, Why They Matter, and How to Fix Them

🎯 Lesson Overview

In software and infrastructure automation, an anti-pattern is a common practice that seems logical but actually creates long-term problems — making your code harder to maintain, harder to understand, and more likely to break. In Puppet, anti-patterns often emerge when teams prioritize short-term speed over long-term stability.

This lesson is based on the Puppet language style guide, which promotes consistent formatting in the Puppet language, giving you a common pattern, design, and style to follow when developing modules. This consistency in code and module structure makes it easier to update and maintain the code.

Addressing anti-patterns means your infrastructure code becomes:

📖

Easier to read and understand

✅

Easier to test and validate

♻️

More reusable across environments

🛡️

Less prone to configuration drift and errors

Anti-Pattern #1

🚫 Hardcoding Data in Puppet Code

What It Is

Hardcoding data means embedding environment-specific values (like IP addresses, package versions, usernames, or file paths) directly inside your Puppet manifests instead of externalizing them.

Why It Matters

Puppet's strength is in reusable code. Code that serves many needs must be configurable: put site-specific information in external configuration data files, rather than in the code itself. When data is baked into your code, changing a value in one environment means you have to touch the code itself — which defeats the purpose of automation.

✅ The Fix: Use Hiera

Using a tool called Hiera, you can separate the data from the code and place it in a centralized location. This allows you to specify guardrails and define known parameters and variations, so that your code is fully testable and you can validate all the edge cases of your parameters.

✘ Anti-Pattern (Hardcoded)
class myapp {
  package { 'myapp':
    ensure => '2.3.1',
  }
}
✔ Best Practice (Hiera)
class myapp (
  String $version = lookup('myapp::version'),
) {
  package { 'myapp':
    ensure => $version,
  }
}
📄 common.yaml
myapp::version: '2.3.1'
Anti-Pattern #2

🚫 Overusing the exec Resource

What It Is

The exec resource runs shell commands. Teams new to Puppet often lean on exec too heavily to accomplish tasks that native Puppet resource types handle far better.

Why It Matters

Any command in an exec resource must be able to run multiple times without causing harm — that is, it must be idempotent. There are three main ways for an exec to be idempotent:

  • The command itself is already idempotent.
  • The exec has an onlyif, unless, or creates attribute, which prevents Puppet from running the command unless some condition is met.
  • The exec has refreshonly => true, which allows Puppet to run the command only when some other resource is changed.

If none of these conditions are met, the exec resource will run on every Puppet agent run, which can cause unintended side effects, system instability, and make your infrastructure non-idempotent.

There's a widespread tendency to use collections of execs to manage resources that aren't covered by an existing resource type. This works fine for simple tasks, but once your exec pile gets complex enough that you really have to think to understand what's happening, you should consider developing a custom resource type instead, as it is much more predictable and maintainable.

✅ The Fix

Use native Puppet resource types (package, service, file, user, etc.) wherever possible. If you must use exec, always include onlyif, unless, or creates to enforce idempotency.

✘ Anti-Pattern
exec { 'install-curl':
  command => '/usr/bin/apt-get install -y curl',
}
✔ Best Practice
package { 'curl':
  ensure => installed,
}
✔ If exec Is Necessary, Make It Idempotent
exec { 'run-migration':
  command => '/opt/app/migrate.sh',
  onlyif  => '/opt/app/needs_migration.sh',
}
Anti-Pattern #3

🚫 Skipping the Roles and Profiles Pattern

What It Is

Teams often assign classes directly to nodes in site.pp or via an ENC without any abstraction layer. This is called "flat" or "direct" classification.

Why It Matters

Without roles and profiles, people typically build system configurations in their node classifier or main manifest, using Hiera to handle tricky inheritance problems. A standard approach is to create a group of similar nodes and assign classes to it, then create child groups with extra classes for nodes that have additional needs. Another common pattern is to put everything in Hiera, using a very large hierarchy that reflects every variation in the infrastructure. But most people find direct building gets difficult to understand and maintain over time.

✅ The Fix: Adopt the Roles and Profiles Method

The roles and profiles method is the most reliable way to build reusable, configurable, and refactorable system configurations. It separates your code into three levels:

  • Component modules — Normal modules that manage one particular technology (e.g., puppetlabs/apache).
  • Profiles — Wrapper classes that use multiple component modules to configure a layered technology stack.
  • Roles — Wrapper classes that use multiple profiles to build a complete system configuration.

A better interface makes hierarchical data easier to use, makes system configuration easier to read, and makes refactoring easier.

✘ Anti-Pattern (Direct Classification)
# site.pp
node 'webserver01.example.com' {
  include apache
  include mysql
  include firewall
}
✔ Best Practice (Roles & Profiles)
# Role
class role::webserver {
  include profile::base
  include profile::apache
  include profile::mysql
}
Anti-Pattern #4

🚫 The params.pp Pattern

What It Is

The params.pp pattern is an older approach where modules used a separate params class with conditional logic and class inheritance to set default parameter values per operating system.

Why It Matters

If you are maintaining old code created prior to Puppet 4.9, you might encounter the use of a params.pp pattern. This pattern makes maintenance and troubleshooting difficult — refactor such code to use the Hiera data-in-modules pattern instead. Class inheritance can have unpredictable effects and makes troubleshooting difficult.

✅ The Fix: Use Data-in-Modules with Hiera

Modules need default values for their class parameters. Before, the preferred way to do this was the params.pp pattern. With Hiera 5, you can use the "data in modules" approach instead.

✘ Anti-Pattern (params.pp)
class my_module (
  String $source = $my_module::params::source,
  String $config = $my_module::params::config,
) inherits my_module::params {
  # body of class
}
✔ Best Practice (Data in Modules)
class my_module (
  String $source,
  String $config,
) {
  # body of class
}
📄 data/common.yaml
my_module::source: 'default source value'
my_module::config: 'default config value'
Anti-Pattern #5

🚫 Duplicate Resource Declarations

What It Is

Declaring the same resource (same type and title) more than once in the same catalog.

Why It Matters

Each resource must be unique; Puppet does not allow you to declare the same resource twice. This is to prevent multiple conflicting values from being declared for the same attribute. Puppet uses the resource title and the name attribute or namevar to identify duplicate resources — if either the title or the name is duplicated within a given resource type, catalog compilation fails.

This is a common mistake when multiple modules or profiles try to manage the same resource independently.

✅ The Fix

To provide the same resource for multiple classes, use a class or a virtual resource to add it to the catalog in multiple places without duplicating it. Extract shared resources into a single shared profile (e.g., profile::base) and include it once, avoiding redundant declarations.

✘ Anti-Pattern (Duplicate Declaration)
# In profile::apache
package { 'curl':
  ensure => installed,
}

# In profile::base (same catalog)
package { 'curl':
  ensure => installed,
  # Duplicate! Catalog compilation will FAIL.
}
✔ Best Practice (Shared Profile)
# profile::base — single source of truth
class profile::base {
  package { 'curl':
    ensure => installed,
  }
}

# Other profiles simply include it:
# include profile::base
Anti-Pattern #6

🚫 Classes That Do Too Much (Monolithic Classes)

What It Is

Creating large, monolithic classes that try to manage too many things at once — essentially a "god class" that handles multiple unrelated concerns.

Why It Matters

When in doubt, err on the side of simplicity. A module should contain related resources that enable it to accomplish a task. If you describe the function of your module and you find yourself using the word "and," consider splitting the module. Have one goal, with all your classes and parameters focused on achieving it.

✅ The Fix: Structure Classes for a Single Purpose

Structure classes and defined types to accomplish one task. Additionally, split your module into public and private classes and defined types where possible.

  • Public classes — contain the parts of the module meant to be configured or customized by the user.
  • Private classes — contain things you do not expect the user to change via parameters.

Separating into public and private classes or defined types helps build reusable and readable code.

✘ Anti-Pattern (Monolithic)
class everything {
  # Manages NTP, Apache, Users,
  # Firewall, SSH — all in one class!
}
✔ Best Practice (Single Purpose)
class profile::ntp { ... }
class profile::apache { ... }
class profile::users { ... }
class profile::firewall { ... }
Anti-Pattern #7

🚫 Putting Resource Declarations in Node Definitions

What It Is

Placing complex logic, resource declarations, or conditional statements directly inside node definitions in site.pp.

Why It Matters

Although node definitions can contain almost any Puppet code, you should use them only to set variables and declare classes. Avoid putting resource declarations, collectors, conditional statements, chaining relationships, and functions in node definitions; all of these belong in classes or defined types.

✅ The Fix

Keep node definitions minimal. Use them only to assign classes or set variables.

✘ Anti-Pattern (Logic in Node Defs)
node 'webserver01.example.com' {
  file { '/etc/myapp.conf':
    ensure  => file,
    content => 'hardcoded content',
  }
  if $facts['os']['family'] == 'RedHat' {
    package { 'httpd':
      ensure => installed,
    }
  }
}
✔ Best Practice (Minimal Node Def)
node 'webserver01.example.com' {
  include role::webserver
}
Anti-Pattern #8

🚫 Ignoring Idempotency

What It Is

Writing Puppet code that doesn't guarantee the same outcome when applied multiple times — meaning it causes changes on every run, even when the system is already in the desired state.

Why It Matters

A key feature of Puppet is its idempotency: the ability to repeatedly apply a manifest to guarantee a desired resource state on a system, with the same results every time. If a given resource is already in the desired state, Puppet performs no actions. If a given resource is not in the desired state, Puppet takes whatever action is necessary to put the resource into the desired state.

Idempotency enables Puppet to simulate resource changes without performing them, and lets you set up configuration management one time, fixing configuration drift without recreating resources from scratch each time Puppet runs.

✅ The Fix

Use native Puppet resource types (which are inherently idempotent), and if using exec, always guard it with onlyif, unless, or creates. Test your manifests with --noop to confirm they don't report spurious changes.
