🔧 What Are Puppet Tasks?

Learn how Bolt tasks let you run single, on-demand actions across your infrastructure — complementing Puppet's declarative model with powerful imperative capabilities when you need them most.

📚 Self-Paced 🎯 Beginner–Intermediate 🛠 Live Code Examples ⏱ ~15 Minutes
Concept

What Are Puppet Tasks?

Bolt tasks are single actions that you can run on target nodes in your infrastructure, allowing you to make as-needed changes to remote systems. You can run tasks with the Puppet Enterprise (PE) orchestrator or with Puppet's standalone task runner, Bolt.

📜

Script-Like, but Better

Bolt tasks are similar to scripts, but they are kept in modules and can have metadata. This allows you to reuse and share them across your organization instead of scattering one-off scripts.

🔄

Run via PE or Bolt

Execute tasks through the Puppet Enterprise orchestrator for enterprise-grade orchestration, or use Bolt as a lightweight, agentless task runner for quick ad-hoc actions.

📦

Module-Based

Tasks live inside Puppet modules alongside classes, plans, and other content. This makes them easy to version, distribute via the Forge, and integrate into existing workflows.

💡 Declarative vs. Imperative

Puppet manifests manage ongoing state (declarative), while tasks handle one-off, imperative actions — making them a powerful complement in real environments. Think of manifests as "this is how things should be" and tasks as "do this right now."

Motivation

💡 Why Do Tasks Matter?

Sometimes you need to do arbitrary tasks in your infrastructure that aren't about enforcing the state of machines. You might need to restart a service, run a troubleshooting script, or get a list of the network connections to a given node. Tasks let you do exactly that.

🔁

Restart Services

Quickly bounce a service on one or many nodes without writing or modifying a Puppet manifest. Ideal for post-deployment restarts or incident response.

🔍

Troubleshoot Servers

Run diagnostic scripts across your fleet to gather logs, check connectivity, or inspect system state — all from a central point.

🌐

Inspect Network Connections

Get a real-time list of network connections to a given node. Tasks give you on-demand visibility without permanent agents or monitoring changes.

⚡

Complement Declarative Code

Tasks fill the gap where Puppet's declarative model isn't the right fit. One-off patches, data migrations, cache flushes — tasks handle the imperative work cleanly.

Deep Dive

⚙️ How Do Tasks Work?

Tasks are executable files stored in Puppet modules. They can be written in any language the target supports, carry JSON metadata for parameter validation, and accept input in multiple ways.

  1. Writing a Task

    You can write tasks in any programming language the targets run, such as Bash, PowerShell, or Python. A task can even be a compiled binary that runs on the target. Place your task in the ./tasks directory of a module and add a metadata file to describe parameters and configure task behavior.

  2. Naming a Task

    Task names are composed of one or two name segments indicating:

    • The name of the module where the task is located.
    • The name of the task file, without the extension.

    For example, in the puppetlabs/mysql module, the sql task is located at ./mysql/tasks/sql.rb, so the task name is mysql::sql.

  3. Task Metadata

    Task metadata files describe task parameters, validate input, and control how Bolt executes the task. Your task must have metadata to be published and shared on the Forge. Specify task metadata in a JSON file with the naming convention <TASKNAME>.json, placed in the module's ./tasks folder alongside your task file.

    JSON — metadata example 
    {
  "description": "Display a custom message",
  "parameters": {
    "message": {
      "description": "The message to display",
      "type": "String"
    }
  }
}

  4. Passing Parameters

    Tasks can receive input as either environment variables, a JSON hash on standard input, or as PowerShell arguments. By default, Bolt submits parameters as both environment variables and as JSON on stdin.

    📌 The PT_ Prefix Convention

    To add a parameter to your task as an environment variable, pass the argument prefixed with PT_. For example, to add a message parameter, read it in your task code as PT_message. When a user runs the task, they specify the value on the command line as message=hello, and the task runner submits the value hello to the PT_message variable.

Hands-On

📝 Example: A Simple Bash Task

Below is a minimal Bash task that reads a parameter called message and echoes it back. This demonstrates the core pattern: receive a parameter via the PT_ environment variable, then act on it.

Bash — tasks/message.sh 
#!/usr/bin/env bash
# A simple Puppet Bolt task that echoes a user-supplied message.
# The parameter "message" is received via the PT_ prefix convention.

echo "Your message is: $PT_message"
🧪 How It Works Step-by-Step

1. The user runs the task and passes message=hello on the command line.
2. Bolt connects to the target node (via SSH or WinRM).
3. Bolt uploads the script and sets the environment variable PT_message=hello.
4. The script executes and outputs: Your message is: hello

Running a Task with Bolt

Use the bolt task run command followed by the fully qualified task name and the target(s). Here's an example that runs the built-in puppet_agent::version task locally:

Shell 
# Run the puppet_agent::version task against localhost
bolt task run puppet_agent::version --targets localhost

# Run our custom message task against a remote node
bolt task run mymodule::message message=hello --targets webserver01.example.com

Module Directory Structure

A well-structured task module looks like this. The task file and its matching JSON metadata live together in the ./tasks directory.

File Structure 
mymodule/
├── metadata.json
├── manifests/
│   └── init.pp
└── tasks/
    ├── message.sh          # The task script (Bash)
    └── message.json        # Task metadata (parameters, description)

🎯 When to Use Tasks vs. Manifests

Use manifests when you want to declare and continuously enforce the desired state of a system (e.g., "package X must always be installed").

Use tasks when you need a one-time imperative action (e.g., "restart service Y right now" or "pull the latest deployment artifact"). They are complementary — not competing — tools in a Puppet-managed environment.

Summary

Key Takeaways

01

Tasks = On-Demand Actions

Bolt tasks are single actions you run on targets. They complement Puppet's declarative manifests by handling imperative, one-off work.

02

Any Language, Any Target

Write tasks in Bash, PowerShell, Python, Ruby, or even ship a compiled binary. If the target can run it, Bolt can deliver it.

03

Metadata Makes Tasks Shareable

JSON metadata describes parameters, validates input, and is required to publish tasks on the Puppet Forge for community reuse.
